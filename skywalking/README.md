# Apache SkyWalking

Apache SkyWalking is an open-source Application Performance Management (APM) system, designed solely for microservices, cloud-native, and container-based architectures. It provides observability capabilities for distributed systems, including tracing, metrics, logging, and event handling.

## üèó Architecture

SkyWalking is logically split into four distinct parts:

1.  **Probes (Agents)**: Collects telemetry data (traces, metrics, logs) from various platforms (Java, Python, Go, Node.js, Service Mesh, etc.) and formats them for the backend.
2.  **Platform Backend (OAP Server)**: The Observability Analysis Platform. It receives data from probes, analyzes it (aggregating metrics, building topology maps), and stores it.
3.  **Storage**: The persistent storage for SkyWalking data. It supports modular backends including **Elasticsearch**, **PostgreSQL**, **MySQL**, **BanyanDB**, and more.
4.  **UI**: A highly customizable web-based interface for visualizing the analyzed data.

## üß© Components

A typical SkyWalking deployment involves several key components:

### Core Components
*   **OAP Server**: Observability Analysis Platform (OAP). The core analysis and processing backend.
*   **Web UI**: The visualization frontend.

### Agents & Probes
SkyWalking supports a wide range of agents for different languages and environments:
*   **[Java Agent](https://skywalking.apache.org/docs/skywalking-java/next/readme/)**: Native tracing/metrics/logging for Java.
*   **[Python Agent](https://skywalking.apache.org/docs/skywalking-python/next/readme/)**: Tracing/metrics/profiling for Python.
*   **Node.js Agent**: For Node.js applications.
*   **Go Agent**: For Golang applications.
*   **[SkyWalking Rover](https://skywalking.apache.org/docs/skywalking-rover/next/readme/)**: A metric collector and profiler powered by eBPF to diagnose CPU and network performance, often used in Kubernetes environments.
*   **[SkyWalking Satellite](https://skywalking.apache.org/docs/skywalking-satellite/next/readme/)**: A lightweight collector/sidecar deployed close to the target system to collect telemetry data and forward it to the OAP server. Useful for edge computing or handling high load.

### Database
*   **[BanyanDB](https://skywalking.apache.org/docs/skywalking-banyandb/next/readme/)**: An observability database native to SkyWalking. It is optimized to ingest, analyze, and store Metrics, Tracing, and Logging data generated by SkyWalking, offering better performance and resource usage than general-purpose databases.

### Kubernetes & Operations
*   **[SkyWalking Cloud on Kubernetes (SWCK)](https://skywalking.apache.org/docs/skywalking-swck/next/readme/)**: A bridge between SkyWalking and Kubernetes. It provides an Operator, custom metrics adapter, and agent injector to simplify deploying and managing SkyWalking on K8s.
*   **[SkyWalking CLI](https://github.com/apache/skywalking-cli)**: A command-line interface for interacting with SkyWalking, useful for operations and querying data.

## üöÄ Installation & Patterns

There are multiple ways to install and deploy Apache SkyWalking depending on your environment.

### 1. Standalone (Binary)
Best for quick testing or ensuring components work locally.
1.  Download the binary tarball from the [official website](https://skywalking.apache.org/downloads/).
2.  Extract the file.
3.  Run `bin/startup.sh`.
    *   This starts both the OAP server and the Web UI.
    *   By default, it uses H2 storage (in-memory/file), which is **not** recommended for production.

### 2. Docker Standalone
Quickly start SkyWalking with Docker.
```bash
# Start OAP
docker run --name oap -d -p 12800:12800 -p 11800:11800 apache/skywalking-oap-server:10.0.0

# Start UI linked to OAP
docker run --name oap-ui -d -p 8080:8080 --link oap:oap -e SW_OAP_ADDRESS=http://oap:12800 apache/skywalking-ui:10.0.0
```

### 3. Kubernetes (Helm Chart)
The recommended way for production deployments.

**Prerequisites:**
*   Kubernetes Cluster
*   Helm 3

**Installation:**
```bash
export SKYWALKING_RELEASE_NAME=skywalking
export SKYWALKING_RELEASE_NAMESPACE=skywalking

# Install with Elasticsearch as storage (default)
helm install "${SKYWALKING_RELEASE_NAME}" \
  oci://registry-1.docker.io/apache/skywalking-helm \
  --version "4.8.0" \
  -n "${SKYWALKING_RELEASE_NAMESPACE}" \
  --create-namespace \
  --set oap.image.tag=10.0.0 \
  --set oap.storageType=elasticsearch \
  --set ui.image.tag=10.0.0
```

### 4. BanyanDB Storage Setup
To use SkyWalking's native database **BanyanDB** for better performance:

```bash
helm install "${SKYWALKING_RELEASE_NAME}" \
  oci://registry-1.docker.io/apache/skywalking-helm \
  --version "4.8.0" \
  -n "${SKYWALKING_RELEASE_NAMESPACE}" \
  --set oap.image.tag=10.0.0 \
  --set oap.storageType=banyandb \
  --set ui.image.tag=10.0.0 \
  --set elasticsearch.enabled=false \
  --set banyandb.enabled=true \
  --set banyandb.image.tag=0.6.0
```
